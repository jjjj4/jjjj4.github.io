AQS的作用

AQS 是一个用来构建锁和同步器的框架，为请求共享资源失败的线程而进入阻塞等待提供了维护以及被唤醒时锁分配的机制。

AQS的原理

核心思想：如果被请求的共享资源空闲，则将当前请求资源的线程设置为有效的工作线程，并且将共享资源设置为锁定状态。如果被请求的共享资源被占用，那么就需要一套线程阻塞等待以及被唤醒时锁分配的机制。

AQS内部维护了一个FIFO的双向链表，获取共享资源失败的线程将会被封装成该链表的一个节点加入到AQS的队列中，并且该线程进入阻塞状态。当获取锁的线程释放锁之后，会在队列中唤醒一个阻塞的节点。

AQS结构

//　头结点，可以理解为当前持有锁的线程
private transient volatile Node head;

//　阻塞的尾节点
private transient volatile Node tail;

//　当前锁的状态，0代表没有被占用，大于0表示有线程持有当前锁
private volatile int state;

// 当前持有独占锁的线程

Node结构

static final class Node {
    //　标志当前节点在共享模式下
    static final Node SHARED = new Node();

    //　标志当前节点在独占模式下
    static final Node EXCLUSIVE = null;

    // === 下面几个常量是　waitStatus　的几个状态 ===
    //　代表此node取消争抢该锁
    static final int CANCELLED = 1;
    //　代表此node的后继节点将被唤醒
    static final int SIGNAL = -1;
    //　代表此node对应的线程正在等某个状态
    static final int CONDITION = -2;
    // ...
    static final int PROPAGATE = -3;


    volatile int waitStatus;

    //　前驱
    volatile Node prev;
    //　后继
    volatile Node next;
    //　当前线程
    volatile Thread thread;
}

ReentrantLock

ReentrantLock在内部通过内部类Sync来管理锁，真正获取锁和释放锁是由Sync的实现类来控制的。
Sync有两个实现类，分别为NonfairSync(非公平锁)和FairSync(公平锁)
在创建ReettrantLock对象时，可传入boolean来指定具体的锁类别

lock()

//　上锁是通过调用 AQS 的 acquire()实现
final void lock() {
    sync.acquire(1);
}

// AQS　的 acquire(int arg)方法
public final void acquire(int arg) {
    //　如果 tryAcquire()成功了返回true,　那么就结束了。含义是：尝试获取锁，成功了也就不必进入队列排队了

    //　如果 tryAcquire()失败了，则需要将当前线程挂起，放入阻塞队列中
    if (!tryAcquire(arg) && acquireQueued(addWaiter(Node.EXCLUSIVE), arg))
        selfInterrupt();
}

//　公平锁 的　tryAcquire(int arg)方法
//　代表是否可以获取到锁
//　返回true的两种情况：1、没有线程在等待锁；　2、重入锁
public final boolean tryAcquire(int acquires) {
    final Thread current = Thread.currentThread();
    int c = getState();
    // c == 0表示没有线程持有锁
    if (c == 0) {
        //　如果排队队列中没有线程，则CAS尝试一下
        if (!hasQueuedPredecessors() && compareAndSetState(0, acquires)) {
            //　拿到锁，标记
            setExclusiveOwnerThread(current);
            return true;
        }
    } else if (current == getExclusiveOwnerThread()) {
        //　当前线程就是持有锁的线程，则代表重入
        int nextc = c + acquires;
        if (nextc < 0)
            throw new Error("Maximum lock count exceeded");
        setState(nextc);
        return true;
    }

    //　没有获得锁，返回false
    return false;
}

//　再看看 addWaiter()：将线程封装成Node,同时进入到队列中
private Node addWaiter(Node mode) {
    Node node = new Node(Thread.currentThread(), mode);
    Node pred = tail;
    if (pred != null) {
        node.prev = pred;
        // CAS的设置到队尾
        if (compareAndSetTail(pred, node)) {
            pred.next = node;
            return node;
        }
    }
    //　如果队列为空　或 CAS失败(有线程竞争入队),　则自旋的方式入队
    enq(node);
    return node;
}

//　采用自旋的方式入队
private Node enq(final Node node) {
    for (;;) {
        Node t = tail;
        if (t == null) { 
            if (compareAndSetHead(new Node()))
                tail = head;
        } else {
            node.prev = t;
            if (compareAndSetTail(t, node)) {
                t.next = node;
                return t;
            }
        }
    }
}

// acquireQueued()
final boolean acquireQueued(final Node node, int arg) {
    boolean failed = true;
    try {
        boolean interrupted = false;
        for (;;) {
            final Node p = node.predecessor();
            //　如果当前节点是阻塞队列的第一个，那么尝试抢一下锁
            if (p == head && tryAcquire(arg)) {
                setHead(node);
                p.next = null; // help GC
                failed = false;
                return interrupted;
            }

            // 没拿到锁，则需要找到一个waitStatus==-1的节点作为前驱，没找到则将前驱节点的waitStatus设置为-1,再次进入
            // parkAndCheckInterrupt()就是调用 LockSupport.park(this)来挂起当前线程
            if (shouldParkAfterFailedAcquire(p, node) &&
                parkAndCheckInterrupt())
                interrupted = true;
        }
    } finally {
        if (failed)
            cancelAcquire(node);
    }
}

unlock()

// 释放锁的操作是通过 AQS 的 release()实现的
public void unlock() {
    sync.release(1);
}

//　释放锁的实现
public final boolean release(int arg) {

    if (tryRelease(arg)) {
        //　如果尝试释放锁成功，则根据阻塞队列有线程在等待，去唤醒它的后继节点
        NOde h = head;
        if (h != null && h.waitStatus != 0)
            //　将当前节点的waitState设置为0,　从后先前找waitStatus <= 0的节点, 将排在最前面的唤醒
            unparkSuccessor(h);
        return true;
    }

    return false;
}

// 尝试释放锁
protected final boolean tryRelease(int releases) {
    int c = getState() - releases;
    if (Thread.currentThread() != getExclusiveOwnerThread())
        throw new IllegalMonitorStateException();
    // 是否完全释放锁
    boolean free = false;
    // 其实就是重入的问题，如果c==0，也就是说没有嵌套锁了，可以释放了，否则还不能释放掉
    if (c == 0) {
        free = true;
        setExclusiveOwnerThread(null);
    }
    setState(c);
    return free;
}

公平锁和非公平锁的对比

  1. 非公平锁在调用 lock 后，首先就会调用 CAS 进行一次抢锁，如果这个时候恰巧锁没有被占用，那么直接就获取到锁返回了。
  2. 非公平锁在 CAS 失败后，和公平锁一样都会进入到 tryAcquire 方法，在 tryAcquire 方法中，如果发现锁这个时候被释放了（state == 0），非公平锁会直接 CAS 抢锁，但是公平锁会判断等待队列是否有线程处于等待状态，如果有则不去抢锁。

AQS中的ConditionObject

Condition机制是为了实现线程的等待和唤醒，只有满足条件的线程才能够被唤醒

Condition 是依赖于 ReentrantLock 的，不管是调用 await 进入等待还是 signal 唤醒，都必须获取到锁才能进行操作

Condition实现生产者-消费者

class ProductorAndConsumer {
    final Lock lock = new ReentrantLock();

    // condition依赖于ReentrantLock产生
    final Condition productor = lock.newCondition();
    final Condition consumer = lock.newCondition();

    final Object[] items = new Object[100];
    int putptr, takeptr, count;

    // 生产
    public void put(Object x) throws InterruptedException {
        lock.lock();

        try {
            // 如果队列已经满了，则生产者进入阻塞
            while (count == items.length) {
                productor.await();
            }

            items[putptr] = x;
            if (++putptr == items.length)
                putptr = 0;
            ++count;
            consumer.signal();      // 生产了新的产品，唤醒消费者消费
        } finally {
            lock.unlock();
        }
    }

    // 消费
    public Object take() throws InterruptedException {
        lock.lock();

        try {
            // 队列为空，消费者进入阻塞状态
            while (count == 0)
                consumer.await();

            Object x = items[takeptr];
            if (++takeptr == items.length)
                takeptr = 0;
            --count;
            productor.signal();     // 消费者消费了，唤醒生产者进行生产
            return x;
        } finally {
            lock.unlock();
        }

    }
}

AQS中的ConditionObject

  • 条件队列和阻塞队列的节点，都是 Node 的实例，因为条件队列的节点是需要转移到阻塞队列中去的；

  • 一个 ReentrantLock 实例可以通过多次调用 newCondition() 来产生多个 Condition 实例，这里对应 condition1 和 condition2。注意，ConditionObject 只有两个属性 firstWaiter 和 lastWaiter；

  • 每个 condition 有一个关联的条件队列，如线程 1 调用 condition1.await() 方法即可将当前线程 1 包装成 Node 后加入到条件队列中，然后阻塞在这里，不继续往下执行，条件队列是一个单向链表；

  • 调用condition1.signal() 触发一次唤醒，此时唤醒的是队头，会将condition1 对应的条件队列的 firstWaiter（队头） 移到阻塞队列的队尾，等待获取锁，获取锁后 await 方法才能返回，继续往下执行。
